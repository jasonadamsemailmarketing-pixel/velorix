1
Lead submits BANT intake form on Tally.so

Structured fields for Budget, Authority, Need and Timeline — no generic message box.

2
Zapier creates Person, Organization and Deal in Pipedrive

Three chained 2-step Zaps on the free tier. BANT data is mapped to the AI Qualification Summary field on Deal creation.

3
Pipedream links Contact and Org to the Deal

A Node.js step searches Pipedrive for the newly created records and links them, solving the free-tier limitation where Zapier cannot pass IDs between Zaps.

4
Hunter.io verifies the lead's email address

Verification status, score, disposable flag and webmail flag are logged as a note on the Deal automatically.

5
Gemini 2.5 Flash scores the lead using BANT

Returns a 1–10 score, qualification summary, recommended next step and tier. Written to three custom Pipedrive fields and logged as a note.

6
Pipedrive native automations route the lead by score

🔥 Hot (8–10): Stage → AI Qualified Lead  |  ⚡ Warm (5–7): Call activity created  |  ❄️ Cold (1–4): Stage → AI Unqualified Lead

Engineering Decisions

Why It's Built This Way

Architecture decisions that reflect real RevOps constraints — not ideal-world assumptions.

Free-Tier Architecture

Zapier free tier limits each Zap to one trigger and one action with no conditional branching. Rather than upgrading, Zapier handles simple data transport (form → CRM) while Pipedream handles multi-step API logic. Each tool does what it does best.

Webhook Timing Fix

The AI Scorer originally fired on Deal Created — before the Person was linked. Gemini scored leads lower because contact details were missing. Switching to Deal Updated (fires after linking completes) pushed scores from 6–7 to 8–9 on strong BANT leads.

Native CRM Routing

Zapier's Paths feature (conditional branching) requires a paid plan. Pipedrive's built-in Workflow Automation handles score-based routing at no extra cost — and keeping conditional logic inside the CRM is cleaner RevOps architecture.

API Key Security

All API keys (Pipedrive, Hunter.io, Gemini) are stored as Pipedream Workspace Environment Variables and referenced via process.env — never hardcoded in deployed workflow code.
